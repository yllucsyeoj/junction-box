# Junction Box API Assessment — L3 Probe Run
**Date:** 2026-05-07
**Dimensions covered:** Discovery L3, Manifest accuracy L3, Pipeline construction L3, Error response consistency L3, Schema consistency L3, Type system L3, Node categories L3, Token efficiency L3, Async / reference mode L3

## Findings

### F-1: `wirable_params` field mismatch in orientation
**Issue:** The orientation paragraph says the catalog returns `has_wirable_params` (boolean). The actual field is `wirable_params` (a list of param port names). No `has_wirable_params` field exists.

**Observed:** `GET /catalog` node objects: `{"name": "fetch", ..., "wirable_params": ["url"]}`. No boolean field present.

**Agent impact:** Any agent that checks `node.has_wirable_params` gets `undefined`. Low severity (agent can inspect `wirable_params.length > 0`) but the orientation is actively misleading.

---

### F-2: `?async=true` is undocumented and executes synchronously
**Issue:** `POST /exec?async=true` appears nowhere in the manifest, gotchas, or quick_start. When called, it executes synchronously and returns a full `status: "complete"` result — identical to regular `/exec`. It does NOT return a pending run_id.

**Observed:**
```
POST /exec?async=true → HTTP 200, {"status": "complete", "result": [...]}
```
This is true even for external network calls (fetch with a slow URL).

**Agent impact:** High confusion. An agent expecting async behavior gets a synchronous result. The correct async mechanism is the `X-Reference: true` header (reference mode), which does work correctly. Two async patterns exist; one is broken/undocumented.

---

### F-3: Failed reference runs don't promote errors to top level
**Issue:** For a successful `GET /runs/:id`, the result is promoted to `.result` (top-level shortcut). For a failed run, `.result` is `null` and top-level `.errors` is also `null`. The actual errors are only at `.response.errors`.

**Observed:**
```json
GET /runs/{failed-id}:
{
  "status": "error",
  "result": null,
  "errors": null,
  "response": {
    "errors": {"src": {"message": "DNS Error...", ...}}
  }
}
```

**Agent impact:** An agent checking `.errors` (by analogy with POST /exec which returns `.errors` directly) sees null and can't diagnose the failure. Must know to check `.response.errors` specifically for reference-mode failures.

---

### F-4: Type mismatches not caught at validation — confusing runtime error
**Issue:** Wiring a node whose `output_type` is incompatible with a destination param's type passes validation and fails at runtime with a misleading error.

**Observed:** `const` (output_type: any) → `fetch` url port (expects string) — no validation error. At runtime:
```json
"errors": {"f1": {"message": "Unsupported input (expected: nothing)", "error_type": "runtime", "expected_type": "nothing"}}
```
The message says `expected: nothing` which refers to `fetch.input_type` (it's a source node), not a type mismatch. An agent reading this message cannot understand what went wrong.

**Agent impact:** Two compounding problems: (1) type mismatches slip through validation undetected; (2) the runtime error message is opaque and uses the wrong frame of reference.

---

### F-5: GET error responses lack `suggestion` field
**Issue:** POST /exec validation errors include a `suggestion` field per-error. GET endpoint 404 errors use a different shape with no `suggestion`.

**Observed:**
```json
GET /defs/nonexistent → {"status": "error", "error": "Unknown node type: \"nonexistent\""}
GET /patch/nonexistent → {"status": "error", "error": "Patch \"nonexistent\" not found."}
```
vs POST /exec 422 errors which include `"suggestion": "..."` per validation item.

**Agent impact:** Inconsistent — agents can't rely on a uniform `suggestion` field across the API. Minor but adds cognitive load.

---

### F-6: Broken edge error says `Edge "undefined"` when edge has no `id`
**Issue:** When an edge has no `id` field and references a nonexistent node, the error message says `Edge "undefined" references destination node "ghost-node"`.

**Observed:**
```json
{"message": "Edge \"undefined\" references destination node \"ghost-node\" which does not exist."}
```

**Agent impact:** Low severity — the `id` field on edges is not required and the patterns confirm omitting it is fine. But seeing `"undefined"` as an edge identifier in an error is confusing — looks like a JavaScript leak.

---

### F-7: Multiple return nodes allowed without warning
**Issue:** A graph with two `return` nodes both wired from the same source runs successfully (returns the first declared return's value). No validation warning is emitted.

**Observed:** `{src→r1, src→r2}` → HTTP 200, `result: <src value>`, no warnings.

**Agent impact:** Low severity — dual returns are almost certainly a mistake, but the validator stays silent.

---

### F-8: Stale node count in prompt.md orientation (minor)
**Issue:** The orientation paragraph in prompt.md says "~7K tokens, 141 nodes". Actual count is 140 (confirmed by GET /catalog, GET /health, and the manifest's own quick_start step 1).

## Clean behaviors

- All 10 `/patterns` examples run exactly as-is — all pass
- Category filtering (`?category=X`) is accurate — no cross-category bleed
- All 140 catalog nodes have corresponding `/defs` entries with matching `output_type`
- `wirable_params` list in catalog matches the `wirable: true` flags in `/defs` params exactly
- Required param validation (422) works for all tested data-source nodes
- Cycle detection returns HTTP 400 with `fatal: "Cycle detected"`
- Isolated disconnected nodes caught at validation (422 with actionable message)
- Reference mode (`X-Reference: true`) returns pending immediately and run completes in background
- Data source nodes tested and functional: youtube-search, rss-feed, market-snapshot, market-screener, market-history, github-repo, reddit-search, web-htmd
- Wired params work end-to-end (const→youtube-search query port, const→market-history ticker port)
- `GET /defs/nonexistent` correctly returns 404
- Cycle → 400; unknown node type → 422 with "Did you mean?" suggestion

## Recommendations

| # | What | Why | Priority |
|---|------|-----|----------|
| R-1 | Update orientation: change `has_wirable_params` → `wirable_params` (list) | Agents using the field name directly will get undefined | High |
| R-2 | Remove or document `?async=true` — it executes synchronously and shouldn't exist as an undocumented param | Confuses any agent that finds it in old code or examples | High |
| R-3 | For failed reference runs, promote `.response.errors` to top-level `.errors` | Parity with POST /exec response; top-level `.errors` is null currently | High |
| R-4 | Improve runtime type mismatch error — current `"expected: nothing"` is uninterpretable | Replace with "cannot wire to source node input port — use a wirable param port instead" | Medium |
| R-5 | Add `suggestion` fields to GET 404 responses | Consistency; agents get suggestions on POST errors but not GET | Medium |
| R-6 | Fix `Edge "undefined"` message — use `(no id)` or similar placeholder | JavaScript `undefined` leaking into error output | Low |
| R-7 | Add validation warning when graph has multiple `return` nodes | Almost always a mistake; silent success is confusing | Low |
