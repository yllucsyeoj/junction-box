# Junction Box - Issues Discovered During Testing

## Session Summary
Testing the Junction Box Pipeline API revealed several architectural and UX issues that confuse the LLM model and lead to silently failing or incorrect behavior.

---

## Issue 1: Multiple Inputs to Single Port Not Supported

**Problem**: When wiring multiple nodes to a single input port, only ONE value is used - the rest are silently ignored.

**Example that silently failed**:
```json
{
  "nodes": [
    {"id": "wiki", "type": "wiki-summary", ...},
    {"id": "yt", "type": "youtube-search", ...},
    {"id": "hn", "type": "hn-search", ...},
    {"id": "combined", "type": "to-json", ...}
  ],
  "edges": [
    {"from": "wiki", "to": "combined"},
    {"from": "yt", "to": "combined"},    // ONLY THIS ONE WINS
    {"from": "hn", "to": "combined"}      // SILENTLY IGNORED
  ]
}
```

**Result**: Only the HN data appeared in the output - YouTube and wiki were lost.

**Root Cause**: Input ports accept a single value, not an array. The system doesn't concatenate/merge multiple inputs.

---

## Issue 2: Append Node Multi-Input Doesn't Work

**Problem**: The `append` node is documented as supporting multiple inputs via `--items` port, but wiring to it produces empty or incorrect results.

**Test case**:
```json
{
  "nodes": [
    {"id": "yt", "type": "youtube-search", ...},
    {"id": "hn", "type": "hn-search", ...},
    {"id": "combined", "type": "append", "params": {"items": "[]"}}
  ],
  "edges": [
    {"from": "yt", "from_port": "output", "to": "combined", "to_port": "input"},
    {"from": "hn", "from_port": "output", "to": "combined", "to_port": "--items"}
  ]
}
```

**Result**: Empty array - multi-input wiring to `append` doesn't function as expected.

---

## Issue 3: Silent Failures - No Validation Error

**Problem**: When multiple nodes are wired to a single-input port, no validation error is thrown. The graph "succeeds" but produces wrong/missing data.

**Current Behavior**: Status returns `"complete"` even when data is silently lost.

**Expected Behavior**: Either:
- Fail validation at submission time: "Port X does not accept multiple inputs"
- OR have nodes explicitly support multiple inputs with documented behavior

---

## Issue 4: Node Schema vs Agent Hints Mismatch

**Problem**: Agent hints describe functionality that doesn't match the actual node schema.

**Example - `map` node**:
- Agent hint says: `expr` for transformation
- Actual schema: `column` (string) + `value` (NUON constant)
- These are completely different - hint is misleading

**Example - `str-concat` node**:
- Hint says: "do NOT use NUON quoting (set prefix to label= not \"label=\")"  
- This is cryptic and the example shows `"prefix": ""` which doesn't clarify

---

## Issue 5: Complex Key Accessors Don't Work

**Problem**: Nested key access like `financials.revenue` doesn't work with `get` node.

**Workaround**: Must chain multiple `get` operations or fetch parent object first.

---

## Recommendations

### High Priority

1. **Add validation for multi-input ports**
   - At graph submission time, detect when multiple nodes wire to a single-input port
   - Either reject with clear error: "Port X accepts only 1 input, 3 provided"
   - OR document that certain nodes DO support multiple inputs and how

2. **Fix or document `append` node multi-input behavior**
   - If `--items` port should support multiple wires, make it work
   - If not supported, remove from schema and document as single-input only

3. **Reconcile node schemas with agent hints**
   - `map` hint says "expr" but schema has "column" + "value"
   - Update hints to match actual functionality or fix schema

### Medium Priority

4. **Document multi-source patterns**
   - If combining multiple sources requires specific node arrangements, document with examples
   - e.g., "To combine 3 sources, use `merge` nodes in a tree pattern"

5. **Add `--multi` flag or array syntax**
   - Consider allowing `{"to_port": ["input", "input", "input"]}` or similar
   - Make multi-input explicit rather than trial-and-error

### Nice to Have

6. **Return partial results warning**
   - If some nodes were skipped due to multi-input issues, warn in response
   - Don't return `"status": "complete"` when data was silently lost
