# Junction Box — New Data Source Nodes Plan

## Summary

31 data sources across 4 phases, ordered by implementation simplicity:

| Phase | Approach | Nodes | Effort |
|-------|----------|-------|--------|
| **1** | RSS convenience wrappers | 5 (bbc, cnn, aljazeera, medium-feed, substack-feed) | ~2 hours |
| **2** | Nu-level HTML scraping + search | 3 (arxiv, google-scholar, bbc-search) | ~2–3 days |
| **3** | Obscura headless browser | 11 + (cnn-search, aljazeera-search, medium-search, congress-search) | ~3–4 days |
| **4** | REST API | 8 (posthog, airtable, supabase, nocodb, huggingface, apple-podcasts, dictionary, github-extras) | ~1 day |

**Phase 2 reality**: Most news/academic sites use JS-rendered search (SPA apps). Only 3 sites had server-rendered HTML suitable for Nu scraping. The rest — cnn-search, aljazeera-search, medium-search, congress-search — moved to Phase 3 (Obscura headless browser). Reuters removed entirely (no public RSS feed).

**Reality: Most news/academic sites use JS-rendered SPAs.** Only 3 of the planned 8 sites had server-rendered HTML suitable for scraping:

| Built | Site | Method | Status |
|-------|------|--------|--------|

| ✓ | **google-scholar** | CrossRef REST API (`api.crossref.org/works`), JSON response | `complete` |
| ✓ | **bbc** | HTML scrape with `query web` CSS selectors | `complete` |

| ✗ | **medium** | Blocks scraping (403) — moved to Phase 3 | — |
| ✗ | **congress** | Requires API key + blocks HTML scraping — moved to Phase 3 | — |

    path: "/works",

## Phase 3 — Obscura Headless Browser Integration

### Sites moved from Phase 2 (failed scraping)

These sites use JS-rendered search (SPA, React, etc.) and require a headless browser:

| Site | Reason | Moved from |
|------|--------|-----------|
| **CNN** `cnn-search` | Search page is JS-rendered (React SPA) | Phase 2 |
| **Al Jazeera** `aljazeera-search` | SPA with client-side rendering | Phase 2 |
| **Medium** `medium-search` | Blocks HTTP scrapers (403), requires browser-like rendering | Phase 2 |
| **Congress.gov** `congress-search` | API requires key, HTML search blocks scrapers (403) | Phase 2 |

### Obscura overview

- **Binary**: 70MB, Rust, zero dependencies (V8 bundled)
- **Memory**: ~30MB per instance
- **Stealth**: Built-in anti-fingerprinting, 3520 tracker domains blocked, `navigator.webdriver = undefined`
- **CLI**: `obscura fetch URL --eval "JS" --dump html/text/links --stealth`
- **Parallel**: `obscura scrape url1 url2 ... --concurrency 25 --eval "JS" --format json`
- **CDP**: `obscura serve --port 9222` for full Puppeteer/Playwright API
- **Page load**: ~85ms (vs ~500ms for headless Chrome)

### Nu primitive pattern

Each Commerce/Social node is a thin Nu wrapper that shells out to `obscura fetch`:

```nu
# Search Amazon for products. Returns a table with title, price, rating, reviews, url.
@category commerce
export def "prim-amazon-search" [
    --query: string = ""    # [wirable][required] Search terms
    --limit: string = "10"  # Max results
]: nothing -> table {
    let query_val = if ($query | str starts-with '"') { try { $query | from json } catch { $query } } else { $query }
    if ($query_val | is-empty) {
        error make {msg: "provide --query with search terms"}
    }
    let url = $"https://www.amazon.com/s?k=($query_val | url encode)"
    let eval_js = $"
        Array.from(document.querySelectorAll('[data-component-type=\"s-search-result\"]'))
            .slice(0, ($limit | into int))
            .map(el => ({
                title: el.querySelector('h2 a span')?.textContent?.trim() || '',
                price: el.querySelector('.a-price .a-offscreen')?.textContent?.trim() || '',
                rating: el.querySelector('.a-icon-alt')?.textContent?.trim() || '',
                reviews: el.querySelector('.a-size-base.s-underline-text')?.textContent?.trim() || '',
                url: el.querySelector('h2 a')?.href || '',
            }))
    "
    let raw = (obscura fetch $url
        --eval $eval_js
        --stealth
        --wait-until networkidle0
        --timeout 30
        | str trim)
    if ($raw | is-empty) { return [] }
    $raw | from json
}
```

### Site-specific JS extraction selectors

Each site gets custom CSS selectors embedded in the Nu primitive. These are the extraction rules per site:

| Site | Search URL pattern | Key selectors |
|------|-------------------|---------------|
| **Amazon** | `amazon.com/s?k=QUERY` | `[data-component-type="s-search-result"]` — `.a-price .a-offscreen` (price), `h2 a span` (title), `.a-icon-alt` (rating), `h2 a[href]` (url) |
| **Walmart** | `walmart.com/search?q=QUERY` | `[data-testid="item-stack"]` or `.mb0` cards — `.w_iUH7` (title), `.w_iUH7 .f2` (price), `.w_iUH7 .f7` (rating) |
| **Ebay** | `ebay.com/sch/i.html?_nkw=QUERY` | `.s-item` — `.s-item__title` (title), `.s-item__price` (price), `.s-item__link[href]` (url). Ebay has good server-rendered HTML; might work as Nu scraper too. |
| **Target** | `target.com/s?searchTerm=QUERY` | `[data-test="product-grid"]` children — product title, price, image. JS-heavy. |
| **Temu** | `temu.com/search_result.html?search_key=QUERY` | Heavy Cloudflare + JS. Will likely need `--wait-until networkidle0` and stealth. Selectors TBD after inspection. |
| **Shein** | `shein.com/pdsearch/QUERY` | JS-rendered search results. Anti-bot protections. Selectors TBD. |
| **AliExpress** | `aliexpress.com/wholesale?SearchText=QUERY` | `.list-item` or `.manhattan-container` — `.item-title` (title), `.price-current` (price), `.rate-num` (rating). Some server-side rendering. |
| **X (Twitter)** | `nitter.net/search?q=QUERY` | Use Nitter mirrors (nitter.net, nitter.cz) for public tweet search without auth. `.timeline-item` — `.tweet-content` (text), `.tweet-date a` (date), `.fullname` (author). |
| **LinkedIn** | `linkedin.com/search/results/people/?keywords=QUERY` | Public profiles only. `.reusable-search__result-container` — name, headline, location. Login-wall for full results; limited to public SERP. |

### Nitter for X/Twitter

X/Twitter requires login for search and aggressively blocks scrapers. Using Nitter mirrors avoids auth entirely. The Nu primitive targets `nitter.net` (or configurable Nitter instance) which renders Twitter data as clean server-side HTML.

```nu

]: nothing -> table {
    # Uses obscura fetch for rendering, or http get if Nitter HTML is clean enough
    ...
}
```

### Timing & error handling

Obscura node timeouts should be higher than API/RSS nodes:
- Default: 30s (set via `--timeout 30`)
- Retries: 2 (execution layer default is 1, bump to 2 for commerce nodes via param annotation)
- Rate limiting: no built-in solution — responsibility of callers to not hammer endpoints

### Risk: anti-bot evolving

Commerce and social sites actively fight scrapers. Obscura's stealth mode helps but isn't bulletproof. Long-term mitigation:
- Nitter mirrors for Twitter (community-run, numerous instances)
- Accept ~60–70% success rate for commerce sites
- Amazon/Target may require proxy rotation at scale
- Temu/Shein have the strongest anti-bot (Cloudflare + custom) — most likely to fail

### API details

| Node | Endpoint | Auth | Notes |
|------|---------|------|-------|
| `posthog-query` | `POST app.posthog.com/api/projects/{id}/query` | `--api-key` (Personal API Key) | Returns insights/trends/funnels as tables |
| `airtable-query` | `GET api.airtable.com/v0/{baseId}/{tableId}` | `--token` (Personal Access Token) | Also supports `--filter-formula` and `--view` params |
| `nocodb-query` | `GET {instance}/api/v2/tables/{id}/records` | `--token` (Auth Token) + `--instance-url` | Self-hosted or cloud |
| `supabase-query` | `GET {project}.supabase.co/rest/v1/{table}` | `--anon-key` + `--project-url` | Supports `--select`, `--filter`, `--order` params |
| `huggingface-search` | `GET huggingface.co/api/models?search=...` | Optional `--token` for gated models | Returns model metadata, downloads, likes |
| `podcast-search` | `GET itunes.apple.com/search?term=...&entity=podcast` | None | iTunes Search API. Returns title, artist, feedUrl, artwork |
| `dictionary-lookup` | `GET api.dictionaryapi.dev/api/v2/entries/en/{word}` | None | Returns definitions, phonetics, examples |

### Note: `database` vs existing `db`

The existing `db/` extension has `db-query` (internal SQLite for historical runs). New `database/` category is for external DB connectors. Keep them separate — internal DB introspection vs. external DB integration.

### Existing GitHub nodes

GitHub already has `github-repo`, `github-commits`, `github-contributors`. If more are needed (issues, PRs, search), add to the existing `extensions/github/` directory.

## Authentication Strategy

| Auth type | Phase | Approach |
|-----------|-------|----------|
| **None** (public RSS, arxiv, dictionary, apple podcasts, huggingface) | 1, 2, 4 | No credentials |
| **API key** (posthog, airtable, supabase, nocodb) | 4 | `--api-key` or `--token` param, user supplies their own |

| **OAuth 2.0** | — | NOT supported. X/LinkedIn native auth is deferred indefinitely |

## Obscura Binary — Future Work

The Obscura binary is currently stored in `bin/` and COPY'd into the Docker image. This works but has issues:

- **Large files**: 77MB (obscura) + 75MB (obscura-worker) exceed GitHub's recommended 50MB limit
- **Multi-arch**: ARM64 Linux binary must be built separately from x86_64. Only ARM64 exists in this repo
- **Manual build**: Binary was built from source in a running ARM64 container, then extracted. Not reproducible without that container

### Options for future improvement

| Option | Pros | Cons |
|--------|------|------|
| **Git LFS** | Tracks large binaries, clean git workflow | Requires LFS setup for all contributors |
| **Docker build step** | Build from source in Dockerfile | ~10min first build; `boring-sys2` needs cmake + libclang + V8; inflates image with build deps |
| **Multi-arch CI build** | Automate via GitHub Actions: build Obscura per arch, push to release, Dockerfile downloads | Requires CI setup + release management |
| **`curl` from GitHub Releases** | No binary in repo, download at build time | Only x86_64-linux prebuilt; no ARM64 Linux release |

### Recommended approach

Use a Docker buildx multi-stage build: first stage builds Obscura from source (caching the `cargo` registry + V8 compile), second stage copies only the binaries. First build is slow (~10min), subsequent builds use cache. Works for both ARM64 and x86_64.
