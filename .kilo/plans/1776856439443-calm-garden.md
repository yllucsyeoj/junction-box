# Fine-tuning a local model for Junction Box graph construction

## Context

Junction Box is working well - LLMs can write correct graph format after a few probes. The system has ~64 node types with a NUON-based graph format (nodes + edges). The vision from `docs/model-api-design.md` is that a model only needs 5 interactions: `GET /defs`, `POST /exec`, `POST /patch`, `GET /patch/:alias`.

`sidekick.md` discusses alternatives including a smaller fine-tuned model as option 4.

## The Proposed Approach

1. **Dataset generation** — Use a capable model (e.g., Claude) to generate text requests paired with working, tested graphs
2. **Fine-tuning** — Train an MLX model (Apple Silicon) to do one task: given a natural language request, produce a working junction-box graph, execute it, return the result
3. **Architecture** — Big model becomes zero-knowledge about junction-box internals; it delegates to the fine-tuned local model via an MCP-style interface

```
Big Model (zero-knowledge) → MCP → Fine-tuned MLX model → Junction Box → Structured data
```

## Hybrid Generation Strategy

The user wants a tiered approach:
1. **Claude for request generation** — its imagination and creativity produces diverse, realistic text requests that only a frontier model can envision
2. **Local model for graph construction** — given the request, construct the working NUON graph (can take time since it's batch processing)
3. **Script saves all intermediate work** — every step of the generation pipeline is logged/cached

This means the dataset generation pipeline is:
```
Claude (request only) → Local model (graph) → Junction Box (validate/execute) → Saved dataset
```

This keeps Claude costs minimal (just text requests, not graph construction) while leveraging frontier creativity for realistic use cases.

### Dataset format

```json
{
  "id": "uuid",
  "request": "Fetch the top 10 posts from r/wallstreetbets about NVDA, then calculate average score",
  "generated_by": "claude",
  "graph": {
    "nodes": [...],
    "edges": [...]
  },
  "graph_model": "phi-3-3.8b",
  "execution_result": { "status": "complete", "result": "..." },
  "node_types_used": ["fetch", "sort", "first", "col_stats", "return"],
  "num_nodes": 5,
  "has_multi_input": false,
  "num_probes": 3,
  "attempts": [
    { "probe": 1, "graph": {...}, "error": "..." },
    { "probe": 2, "graph": {...}, "error": "..." },
    { "probe": 3, "graph": {...}, "status": "complete" }
  ]
}
```

The `attempts` field captures the iteration history — this is valuable for understanding what the model gets wrong and also for training error recovery patterns.

## Fine-tuning Approach

### Model choice for MLX

Apple Silicon MLX fine-tuning supports:
- Llama 3 8B — fits in 32GB RAM, good quality
- Phi-3 3.8B — smaller, faster, decent quality

**Recommendation:** Since Phi-3 3.8B is already being evaluated for generation, use the same model for fine-tuning. This keeps the footprint consistent and the model familiar with the graph format.

### Training approach

1. **Base model:** Pre-trained model via MLX
2. **Task formatting:** Each example is `<request> → <NUON graph>`
3. **LoRA fine-tuning:** Only train adapter weights — not full model
4. **Evaluation:** Run generated graphs through junction-box validation and execution

### Training configuration

- **Epochs:** 3-5 (small dataset, avoid overfitting)
- **Learning rate:** ~1e-4 with cosine decay
- **Batch size:** 1-4 (MLX RAM constraints)
- **LoRA rank:** 8-16 (sufficient for this narrow task)
