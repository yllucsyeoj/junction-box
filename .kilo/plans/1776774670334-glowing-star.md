# Plan: Docker Setup Improvements for GoNude

## Current State Analysis

**Dockerfile Issues Identified:**

1. **No multi-stage build** - Image includes build tools (Rust, cargo, build-essential) unnecessarily
2. **Heavy base image** - `debian:bookworm-slim` at ~80MB, but still installs full build toolchain
3. **nu_plugin_htmd built from source** - Clones from GitHub at HEAD, no version pin, could break if repo changes
4. **Layer caching not optimized** - COPY commands for source files come before dependency install
5. **No explicit version pinning** - Rust, Bun, and nushell versions could drift
6. **Large build context risk** - .dockerignore exists but doesn't exclude some potentially large files

**docker-compose.yml Assessment:**
- Looks standard and functional
- Uses named volume for persistence
- Has host.docker.internal access for local development

## Proposed Changes

### 1. Multi-Stage Build (High Priority)

Split into two stages:
- **Build stage**: Contains all build tools, compiles nu_plugin_htmd
- **Runtime stage**: Slim image with only runtime dependencies

```dockerfile
# Stage 1: Build
FROM debian:bookworm-slim AS builder
# ... build tools, compile nu_plugin_htmd ...

# Stage 2: Runtime
FROM debian:bookworm-slim
# Only runtime deps: curl, ca-certificates
# Copy only built artifacts
```

**Benefit**: Final image ~150MB smaller, no build tools exposed

### 2. Use Pre-built nu_plugin_htmd Binary from Sibling Repo (High Priority)

Current: Clones and builds nu_plugin_htmd from source at HEAD (brittle, slow)

Since nu_plugin_htmd is in a sibling directory (`../nu_plugin_htmd`), we can:
1. Build the Linux binary locally (cross-compile or Docker)
2. The GoNude Dockerfile will COPY the pre-built binary instead of cloning and building

**Benefits**:
- No build-time compilation in GoNude container (faster builds)
- No cargo/rust needed in GoNude image
- Reproducible - binary is built once, promoted through environments
- Simpler Dockerfile

**Build approach**:
```bash
# Option A: Cross-compile from macOS (if cargo supports)
cd ../nu_plugin_htmd
cargo build --release --target x86_64-unknown-linux-gnu

# Option B: Build in Docker builder stage
docker run --rm -v $(pwd):/src -w /src rust:latest \
  cargo build --release --target x86_64-unknown-linux-musl
```

**Files needed**:
- `../nu_plugin_htmd/target/x86_64-unknown-linux-musl/release/nu_plugin_htmd`
- Or for arm64: `../nu_plugin_htmd/target/aarch64-unknown-linux-musl/release/nu_plugin_htmd`

**Approach for getting binary into Docker build**:
- **Recommended**: Copy binary to `bin/nu_plugin_htmd` in GoNude repo before build, then `COPY bin/nu_plugin_htmd /usr/local/bin/` in Dockerfile. Add `bin/` to .gitignore for host platform.
- Alternative: Build in Docker builder stage and pass as artifact between stages

### 3. Reorder Layers for Better Caching (Medium)

**Problem**: Any file change invalidates subsequent layers

**Optimized order**:
1. Install system deps (rarely changes)
2. Install Nushell (version pinned)
3. COPY server/package.json → bun install (only when deps change)
4. COPY primitives.nu, extensions/ (changed less often than server code)
5. COPY pre-built nu_plugin_htmd binary
6. COPY server source (changes most frequently)
7. Final runtime stage: only COPY artifacts, no build tools

### 4. Add Build Arguments for Version Control (Medium)

```dockerfile
ARG NU_VERSION=0.111.0
ARG BUN_VERSION=1.x.x
ARG RUST_VERSION=1.85.0
```

### 5. Healthcheck Improvement (Low)

Current: Uses curl to /health

Consider: Use a proper Nu script for healthcheck, or add a `/health` endpoint that validates more (DB connection, plugins loaded)
