# Fix Type System Clarity for Agent Patch Construction

## Problem

An agent cannot reliably construct a working patch in 2-3 tries because:

1. **Type checking only happens at the `input` port** — validate.ts:224 only checks `edge.to_port === 'input'`. Wirable param ports (e.g. `merge --with`, `items --input`) skip type compatibility validation entirely. Wrong connections only fail at Nu runtime with opaque "type mismatch" or "Input type not supported" messages.

2. **Runtime errors lack context** — `normalizeNuError()` strips the useful Nu error detail (file:line, got/expected types). The final message is just "Input type not supported" with no hint about which node or which type was expected.

3. **`map`/`update` NUON param parsing fails silently** — The `value` param (format: `nuon`) fails with "error when loading nuon text" on any value. The agent can't distinguish whether it's a quoting issue, a type issue, or a feature bug.

4. **`summarize` only accepts one op** — `ops: "sum,sum"` fails validation with "invalid value" but doesn't say "only one op per call". Agent wastes tries trying to pass multiple ops.

5. **`merge` only accepts records** — Wiring an `int` (from `count`) into `merge --with` fails at runtime with "Input type not supported" instead of validation-time type mismatch.

6. **Date comparison returns "Can't convert to float"** — Filter on `transaction_date` with string comparison fails with a float conversion error, not a date type error. The `filter` node does string comparison (documented in `param_formats: {value: "plain"}`) but the column type (`datetime`) is not compatible with the `>` operator in the engine.

7. **No way to know what `items` expects as input** — `items` is described as "convert record to table" but wiring multiple things to it wasn't caught at validation. There's no `port_types` entry for `items` to tell the agent its input type.

## Root Cause

The execution engine (Nu subprocess) handles type mismatches at runtime, and error normalization strips the useful parts. The validation layer only covers the `input` port, not wirable param ports.

## Plan

### Phase 1: Fix Validation to Cover Wirable Param Ports

**Files:** `server/validate.ts`

Currently validation only checks type compatibility for `edge.to_port === 'input'` (line 224-239). Extend it to cover wirable param ports too.

**Change:** After the `input` port type check in validate.ts, add a parallel check for edges wired to named wirable params. Use `port_types` from `PRIMITIVE_META` (if defined) to determine expected input type; fall back to `param.type` from spec.ts.

```typescript
// For edges wired to a named param port (not 'input')
if (edge.to_port !== 'input') {
  const paramSpec = toSpec.params.find(p => p.name === edge.to_port)
  if (paramSpec?.wirable && spec.port_types?.[edge.to_port]) {
    const fromSpec = specMap.get(nodeMap.get(edge.from)!.type)
    const expectedType = spec.port_types[edge.to_port]  // e.g. record, table, list
    if (fromSpec && !typesCompatible(expectedType, fromSpec.output_type)) {
      errors.push({ error_type: 'type_mismatch', message: `...expects ${expectedType} but ${fromSpec.name} outputs ${fromSpec.output_type}`, ...})
    }
  }
}
```

This would catch `count(int) → merge(record)` at validation time, with a clear error message.

**Fixes:** Problems #1, #5

### Phase 2: Improve Runtime Error Messages

**File:** `server/execute.ts` — `normalizeNuError()`

Current output: `message: "Input type not supported."` with no expected/got type.
Current: it already tries to extract `got_type` from pipeline_mismatch errors, but the final message doesn't always include it.

**Changes:**

1. When error_type is `type_mismatch`, always include `expected_type` and `got_type` in the SSE event response (already partially done but often lost due to the try/catch wrapper stripping).
2. For `merge` (record-only input), add a specific hint: "merge expects record input — count outputs int. Use col_stats or summarize instead."
3. Add a registry of common misconfiguration hints keyed by node type + error pattern, so specific nodes get specific suggestions (e.g., `summarize` → "only one op per call; chain multiple summarize nodes instead").

**Fixes:** Problems #2, #4, #5

---

### Phase 3: Fix `map`/`update` NUON Value Parsing

**File:** `primitives.nu` — `prim-map` and `prim-update`

The `value` param uses `$value | from nuon`. If the value is passed via environment variable (wired or JSON-encoded), it arrives as a JSON string (e.g. `"\"MSFT\""` or `"MSFT"`). The `from nuon` fails silently or with a confusing error.

**The actual flow:**
- Static param: `--value "MSFT"` → `"MSFT" | from nuon` → works
- Wired param: env var `GONUDE_PARAM_VALUE` = `"\"MSFT\""` (JSON-encoded string) → `'"MSFT"' | from nuon` → tries to parse literal string `'"MSFT"'` as NUON, which is invalid

**Fix:** In `prim-map` and `prim-update`, add the standard wired-param unwrap pattern:
```nushell
let val = if ($value | str starts-with '"') { try { $value | from json } catch { $value } } else { $value }
let resolved = $val | from nuon
```

This handles both static and wired values correctly.

**Fixes:** Problem #3

---

### Phase 4: Fix `filter` Date/datetime Column Comparison

**File:** `primitives.nu` — `prim-filter`

The `>` and `<` operators in Nushell don't work on datetime columns. When `transaction_date > "2026-01-29"` runs, Nu tries to compare datetime to string and fails with "Can't convert to float".

**Fix options:**

**Option A (preferred):** Add datetime-aware comparison in `prim-filter`. Detect if the column contains datetime values and convert the comparison string to datetime before comparing:
```nushell
let col_vals = ($in | get $column)
let is_datetime = ($col_vals | first | get $column | describe | str contains 'datetime')
if $is_datetime and $op in ["==", "!=", ">", "<"] {
    let date_val = ($value | parse "%Y-%m-%d" | get 0 | into datetime)
    # do datetime comparison
} else {
    # existing string comparison
}
```

**Option B:** Document the limitation clearly in the agent_hint and add a `date_filter` node that handles datetime ranges explicitly (e.g. `start_date`, `end_date` params).

**Decision:** Option A is cleaner — one node handles all comparison types correctly.

**Fixes:** Problem #6

---

### Phase 5: Fix `items` Port Type Annotation

**File:** `primitives.nu` — `items` PRIMITIVE_META entry + `server/spec.ts`

`items` currently has no `port_types` in PRIMITIVE_META. It should be:
```nushell
items: {..., port_types: {input: "record"}}
```

So the spec exposes it correctly and validation can check type compatibility. Also verify `group_agg` input type is correctly documented — `group_agg` takes `record` (output of `group_by`).

**Fixes:** Problem #1, #7

---

### Phase 6: Fix `summarize` Multi-Op Documentation

**No code change needed.** The `param_options` for `ops` already lists the valid values. The error message from validation already says "Valid values: avg, sum, min, max, count" which is correct — the issue is the agent doesn't know that only one op is allowed per call (not a comma-separated list).

**Fix:** Update `agent_hint` for `summarize` to explicitly say: "Single operation only — pass one of avg, sum, min, max, count. To get multiple aggregations, chain separate summarize nodes."

**Fixes:** Problem #4

---

### Phase 7: Add Type Info to `GET /defs/:type`

**File:** `server/index.ts` and `server/spec.ts`

Currently `GET /defs/:type` returns input_type, output_type, params, ports. But when a wirable param has a `port_types` entry (like `merge --with: record`), that type info isn't surfaced in a way the agent can easily discover.

**Change:** In the defs response, for each port (input/output/wirable), include the type. For named wirable ports, use `port_types` if available. Example:

```json
{
  "name": "merge",
  "input_type": "record",
  "output_type": "record",
  "ports": {
    "inputs": [{"name": "input", "type": "record", "role": "param"}],
    "wirable_params": [
      {"name": "with", "type": "record", "description": "NUON record to merge"}
    ]
  }
}
```

This lets the agent query one endpoint and know that `merge --with` expects `record`, not `int`.

**Fixes:** Problems #1, #5, #7
